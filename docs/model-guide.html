HL7v2 ADT Models for PClean

Probabilistic data cleaning models derived from the HL7 specification

New to HL7? Read the HL7v2 primer first.
What This Is

A PClean (Julia DSL) model for HL7v2 ADT^A08 (Patient Update) messages, derived entirely from the HL7 v2plus specification (2021Jan edition). No institutional data was used — all priors are uninformative/uniform.

PClean performs entity resolution: given many dirty ADT messages, it infers that records with similar patient names, IDs, and addresses refer to the same latent Patient entity, and recovers the clean values.

Model Structure
Facility — sending/receiving system (from MSH segment) Patient — the person (from PID + PD1 segments) NextOfKin — emergency contact (from NK1 segment) Visit — clinical encounter (from PV1 + PV2 segments) InsurancePlan — coverage info (from IN1 segment) ADT_A08_Record — one observed message references all entities above, with AddTypos corruption on each field
How PClean’s entity model works

PClean distinguishes between latent entities (the true, clean real-world things) and observed records (the dirty data you actually see).

Multiple observed records can reference the same latent entity. For example, 50 ADT messages about the same patient all reference one Patient entity — even though each message spells the name slightly differently. PClean’s inference recovers:

  • The clean attribute values for each entity
  • Which records belong to which entities (entity resolution / deduplication)

Running the Model

Requires PClean.jl:

using Pkg
Pkg.add(url="https://github.com/probcomp/PClean")
include("adt_a08.jl")

To run inference, you need a CSV file with columns matching the query mapping (PID_PatientName, PV1_PatientClass, etc.) — one row per message. The inference section at the bottom of adt_a08.jl shows how to load data, configure the sampler, and evaluate results.

Why ADT^A08

ADT^A08 (Patient Update) is the richest ADT message type — 8 segments covering demographics, next of kin, visit, and insurance. ADT^A01 (Admit) and ADT^A04 (Register) share the same segment composition, so this model works for those too.

We chose a single message type because PClean requires a fixed schema. It cannot model variable-length repeating segments (e.g. multiple NK1 or IN1 per message) or condition segment presence on message type. A model that handles multiple message types or variable repetition would require a more expressive PPL.
