HL7v2 ADT Models for PClean

Probabilistic data cleaning models derived from the HL7 specification

New to HL7? Read the HL7v2 primer first.
What This Is

A PClean (Julia DSL) model for HL7v2 ADT^A08 (Patient Update) messages, derived entirely from the HL7 v2plus specification (2021Jan edition). No institutional data was used — all priors are uninformative/uniform.

PClean performs entity resolution: given many dirty ADT messages, it infers that records with similar patient names, IDs, and addresses refer to the same latent Patient entity, and recovers the clean values.

Model Structure
Facility — sending/receiving system (from MSH segment) Patient — the person (from PID + PD1 segments) NextOfKin — emergency contact (from NK1 segment) Visit — clinical encounter (from PV1 + PV2 segments) InsurancePlan — coverage info (from IN1 segment) ADT_A08_Record — one observed message references all entities above, with AddTypos corruption on each field
How PClean’s entity model works

PClean distinguishes between latent entities (the true, clean real-world things) and observed records (the dirty data you actually see).

Multiple observed records can reference the same latent entity. For example, 50 ADT messages about the same patient all reference one Patient entity — even though each message spells the name slightly differently. PClean’s inference recovers:

  • The clean attribute values for each entity
  • Which records belong to which entities (entity resolution / deduplication)

Running the Model

Requires PClean.jl:

using Pkg
Pkg.add(url="https://github.com/probcomp/PClean")
include("adt_a08.jl")

To run inference, you need a CSV file with columns matching the query mapping (PID_PatientName, PV1_PatientClass, etc.) — one row per message. The inference section at the bottom of adt_a08.jl shows how to load data, configure the sampler, and evaluate results.

What PClean Cannot Model (and PLUCK Can)

This ADT model works within PClean’s constraints, but several real-world HL7 properties fall outside what PClean can express. These are exactly the properties that PLUCK’s recursive algebraic data types and exact inference are designed to handle.

1. Variable-length repeating segments

Real HL7 messages have segments that repeat a variable number of times:

SegmentCardinalityTypical range
NK1 (Next of Kin)0..*0–3
IN1 (Insurance)0..*0–2
OBX (Observations)1..*1–288

PClean requires fixed cardinality — we model exactly 1 NK1 and 1 IN1. PLUCK’s recursive list types can express the variable-length structure:

(define (generate-nk1-list)
  (if (flip 0.3)
    (Cons (generate-nk1) (generate-nk1-list))
    (Nil)))
2. Structural variants (variable field counts)

PV1 has 6 known structural variants in real data:

FieldsFrequency
19 fields47.4%
52 fields41.0%
45 fields9.3%
44 fields2.3%
39, 49 fields<0.01%

PClean models one fixed schema. PLUCK can model the field count itself as a random variable — the number of fields present becomes part of the probabilistic program’s output.

3. Message-type-dependent segment composition

Different ADT event types include different segments:

ADT^A03 (Discharge): MSH  EVN  PID       PV1  PV2
ADT^A08 (Update):    MSH  EVN  PID  PD1  NK1  PV1  PV2  IN1

This requires conditioning segment presence on message type — a structural choice that PClean’s fixed-schema model cannot express. In PLUCK, message type would be a random variable that controls which branches of a recursive generator fire.

4. Cross-segment dependencies

In ORU^R01 (lab results), the value of OBR.universal_service_id (which test was ordered) constrains which OBX observations can appear and what their valid ranges are. These cross-segment correlations require conditional distributions that PClean’s class-reference model doesn’t support.
