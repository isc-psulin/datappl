HL7v2 Primer

For probabilistic programming researchers

What is HL7v2?

HL7v2 is a messaging standard for exchanging clinical data between healthcare IT systems. When a patient is admitted, discharged, gets lab work, or has insurance updated, the hospital’s systems exchange HL7v2 messages. It has been the dominant standard since the late 1980s. A single mid-size hospital generates millions of these messages per year.

Messages and Types

An HL7v2 message is a plain-text record of something that happened in a hospital — a patient was admitted, a lab result came back, an appointment was scheduled. Each message has a type that says what kind of event it represents.

There are many message types. The major categories are:

  • ADT (Admit, Discharge, Transfer) — patient movement and demographics
  • ORU (Observation Result) — lab results
  • SIU (Scheduling) — appointments
  • ORM (Order) — clinical orders

Within each category there are specific event types. For example, ADT alone has dozens: A01 (admit), A02 (transfer), A03 (discharge), A04 (register), A08 (update demographics), and many more.

In this repo we model ADT^A08 (patient demographic update). We chose it because it uses a fixed set of 8 segments covering demographics, next of kin, visit, and insurance — which fits PClean’s fixed-schema requirement. Other message types have variable-length repeating groups that PClean cannot express.

Structure of a Message

A message is made up of segments, one per line. Each segment starts with a three-letter code that identifies what kind of data it carries. Within a segment, fields are separated by | (pipe), and sub-components within a field are separated by ^ (caret).

This message has 8 segments. The first line (MSH) is the message header — it says who sent the message, when, and what type it is. PID carries the patient’s name, date of birth, and address. PV1 describes the visit (location, attending doctor). And so on.

Fields in detail

Fields are positional — their meaning is determined by their position within the segment. For example, in a PID segment:

Empty fields (consecutive ||) are common. The HL7 spec defines what each position means for every segment type.

Sub-components in detail

Some fields have internal structure, using ^ as separator:

Field valueMeaning
SMITH^JOHN^A last name ^ first name ^ middle initial
MR12345^^^BMC^MR ID ^ check digit ^ scheme ^ assigning authority ^ type code
4N^415^1^^^BMC unit ^ room ^ bed ^ (3 empty) ^ facility

This gives HL7v2 a hierarchical structure: message → segments → fields → sub-components.
