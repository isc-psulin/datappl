HL7v2 Primer

For probabilistic programming researchers

What is HL7v2?

HL7v2 is the dominant messaging standard for exchanging clinical data between healthcare IT systems. When a patient is admitted, discharged, gets lab work, or has insurance updated, the hospital's systems exchange HL7v2 messages.

Designed in the late 1980s, it is the ASCII serial protocol of healthcare — simple, ubiquitous, and full of quirks from 35+ years of real-world deployment.

Scale: A single mid-size hospital generates millions of HL7v2 messages per year. The dataset we work with contains ~2 million messages from routine operations.

Message Types

Every HL7v2 message has a type, identified by the MSH-9 field. The format is TYPE^EVENT. The type tells you the clinical domain; the event tells you what happened.

ADT — Admit, Discharge, Transfer

Tracks patient movement through the healthcare facility. The most common message category.

MessageEventWhen it fires
ADT^A01AdmitPatient admitted to inpatient bed
ADT^A02TransferPatient moved to different unit/bed
ADT^A03DischargePatient leaves the facility
ADT^A04RegisterPatient registers (outpatient, ED)
ADT^A08UpdateAny update to patient demographics
ADT^A11Cancel admitUndo an A01
ADT^A13Cancel dischargeUndo an A03
ADT^A34MergeTwo patient records merged into one
Which segments appear in each ADT type? 
ADT^A01 (Admit):     MSH  EVN  PID  PD1  NK1  PV1  PV2  IN1
ADT^A03 (Discharge): MSH  EVN  PID            PV1  PV2
ADT^A04 (Register):  MSH  EVN  PID  PD1  NK1  PV1  PV2  IN1
ADT^A08 (Update):    MSH  EVN  PID  PD1  NK1  PV1  PV2  IN1

All share MSH + EVN + PID + PV1. The optional segments (PD1, NK1, PV2, IN1) vary by event type. NK1 and IN1 can repeat (0..* per message).

ORU — Observation Result (Lab Results)

Carries lab test orders and their results. The most structurally complex message type.

ORU^R01: MSH  PID  PV1  [ORC  OBR  OBX  OBX  OBX ...]
                         |---- repeating group ----|

The [ORC OBR OBX...] group repeats — one OBR per test ordered, with 1–288 OBX results per OBR. This variable-length repetition is a key structural challenge for probabilistic modeling.

SIU — Scheduling

SIU^S12 (New appointment): MSH  SCH  PID  RGS  AIL  AIP
Message Structure

An HL7v2 message is a sequence of segments, each on its own line, with fields separated by | (pipe). Here is an example ADT^A08 message:

MSH|^~\&|EPIC|BMC|||20170614120000||ADT^A08|MSG00001|P|2.5.1 EVN|A08|20170614120000 PID|1||MR12345^^^BMC^MR||SMITH^JOHN^A||19800315|M|||123 MAIN ST^^SPRINGFIELD^MA^01103 PD1|||BMC PRIMARY CARE^^^^||||||||N NK1|1|SMITH^JANE|SPO|456 MAIN ST^^SPRINGFIELD^MA^01103 PV1|1|I|4N^415^1^^^BMC||||1234567890^JONES^ROBERT^J^^^MD|||MED||||1|||1234567890^JONES^ROBERT^J^^^MD PV2||||||||||||||||||||||||||||||||||||||||||||||||20170620 IN1|1||BCBS001|BLUE CROSS BLUE SHIELD|||||||||||SMITH^JOHN^A|SEL|19800315

Reading top to bottom: the three-letter prefix on each line is the segment type. Within each line, | separates fields, and ^ separates sub-components within a field.

Segments

Each segment type carries a specific category of clinical information. A message is composed of segments in a defined order.

Fields within a segment

Fields are positional within a segment. For example, in a PID segment:

PID|1||MR12345^^^BMC^MR||SMITH^JOHN^A||19800315|M
PositionField NameValue
PID-1Set ID1
PID-2Patient ID(empty — withdrawn field)
PID-3Patient Identifier ListMR12345^^^BMC^MR

PID-5Patient NameSMITH^JOHN^A

The HL7 spec defines the meaning and data type of every field position for every segment. PID has up to 40 defined fields; PV1 has up to 52.

Sub-components within a field

Some fields have internal structure, using ^ as separator:

Field valueMeaning
SMITH^JOHN^A last name ^ first name ^ middle initial
MR12345^^^BMC^MR ID ^ check digit ^ scheme ^ assigning authority ^ type code
4N^415^1^^^BMC unit ^ room ^ bed ^ (3 empty) ^ facility

This recursive structure (message → segments → fields → components → sub-components) makes HL7v2 a natural fit for hierarchical probabilistic models.

Why This is a Data Cleaning Problem

In theory, HL7 messages follow the spec perfectly. In practice, they don't.

Character-level corruption

Names, addresses, and IDs accumulate typos as data passes between systems:

SMITH^JOHN    (original)
SMTIH^JOHN    (transposition)
SMITH^JON     (deletion)
SMITH^JOHHN   (insertion)

These are the same patient, observed through noisy channels. Each HL7 message is a potentially corrupted snapshot of the true patient identity.

Structural variation

The HL7 spec defines a fixed number of fields per segment (e.g. PV1 has 52), but implementations often send fewer fields, truncating trailing empty ones. The number of fields present can vary across sending systems, creating structural “dialects” of the same segment type.

Missing and optional data

Fields are optional at multiple levels:

  • A segment may be absent from the message entirely (PD1, NK1, IN1 are optional in ADT)
  • A field within a present segment may be empty (consecutive || delimiters)
  • Sub-components within a field may be missing (consecutive ^^)

The optionality structure depends on the message type, the sending system, and institutional conventions.

Entity resolution

The same patient appears in thousands of messages over years of care. Each message carries a potentially different (corrupted) snapshot of their demographics. The core inference problem: given many noisy observations, recover the true patient identity and link records that refer to the same person.

Vocabulary drift

Code tables (sex, race, marital status) are supposed to use HL7-defined values, but implementations often use local codes, abbreviations, or free text where coded values are expected. A “sex” field might contain M, MALE, 1, or Male depending on the source system.

Why This Matters for PPL Research

HL7v2 is a real-world domain with properties that align with open problems in probabilistic programming:

PropertyPPL challenge
Known generative structure (the spec defines the grammar) The model is derivable from the specification — not hand-crafted per dataset
Massive scale (millions of messages per institution) Inference must be tractable on large datasets
Multiple levels of corruption (character, structural, semantic) Requires composing noise models at different abstraction levels
Natural entity resolution (same patient across many messages) Latent entity inference with an unknown number of entities (CRP/DPM)
Recursive/variable-length structure (repeating segment groups) Recursive algebraic data types and geometric stopping distributions
346+ segment types in the HL7 spec Model generation from formal specifications (meta-programming)

It is a natural testbed for probabilistic programs that need to handle structured data with both value-level noise and structural uncertainty.
