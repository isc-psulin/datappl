HL7v2 Primer

For probabilistic programming researchers

What is HL7v2?

HL7v2 is a messaging standard for exchanging clinical data between healthcare IT systems. When a patient is admitted, discharged, gets lab work, or has insurance updated, the hospital’s systems exchange HL7v2 messages. It has been the dominant standard since the late 1980s — simple, ubiquitous, and full of quirks from 35+ years of real-world deployment.

A single mid-size hospital generates millions of HL7v2 messages per year.

How a Message is Structured

An HL7v2 message is plain text. It is made up of segments, one per line. Each segment starts with a three-letter code that identifies what kind of data it carries. Within a segment, fields are separated by | (pipe), and sub-components within a field are separated by ^ (caret).

Here is an example ADT^A08 (patient update) message:

MSH|^~\&|EPIC|BMC|||20170614120000||ADT^A08|MSG00001|P|2.5.1 EVN|A08|20170614120000 PID|1||MR12345^^^BMC^MR||SMITH^JOHN^A||19800315|M|||123 MAIN ST^^SPRINGFIELD^MA^01103 PD1|||BMC PRIMARY CARE^^^^||||||||N NK1|1|SMITH^JANE|SPO|456 MAIN ST^^SPRINGFIELD^MA^01103 PV1|1|I|4N^415^1^^^BMC||||1234567890^JONES^ROBERT^J^^^MD|||MED||||1|||1234567890^JONES^ROBERT^J^^^MD PV2||||||||||||||||||||||||||||||||||||||||||||||||20170620 IN1|1||BCBS001|BLUE CROSS BLUE SHIELD|||||||||||SMITH^JOHN^A|SEL|19800315

Each line is a segment. The first line (MSH) is the message header — it contains metadata like who sent the message and what type it is. The PID line carries patient identification (name, date of birth, address). The PV1 line describes the patient’s visit (location, attending doctor).

Fields in detail

Fields are positional — their meaning is determined by their position within the segment. For example, in a PID segment:

Empty fields (consecutive ||) are common. The HL7 spec defines what each position means for every segment type.

Sub-components in detail

Some fields have internal structure, using ^ as separator:

Field valueMeaning
SMITH^JOHN^A last name ^ first name ^ middle initial
MR12345^^^BMC^MR ID ^ check digit ^ scheme ^ assigning authority ^ type code
4N^415^1^^^BMC unit ^ room ^ bed ^ (3 empty) ^ facility

This gives HL7v2 a hierarchical structure: message → segments → fields → sub-components.

Segment Types

Each three-letter code identifies a segment type. Here are the ones used in ADT messages:

SegmentNameWhat it carries
MSHMessage HeaderMetadata: who sent it, when, what version, what type of message
EVNEvent TypeWhat triggered this message (admit, discharge, update, etc.)
PIDPatient IdentificationName, DOB, sex, race, address, account numbers
PD1Patient DemographicsLiving situation, primary care facility, living will

Other message types use additional segments like OBR (lab orders), OBX (lab results), and ORC (order control).

Message Types

Every HL7v2 message has a type that describes what clinical event triggered it. The type is carried in the MSH segment (the first line of every message). It has two parts: a category and an event. For example, ADT^A08 means category ADT (patient administration), event A08 (update).

ADT — Admit, Discharge, Transfer

The most common category. Tracks patient movement through the facility:

MessageEventWhen it fires
ADT^A01AdmitPatient admitted to inpatient bed
ADT^A02TransferPatient moved to different unit/bed
ADT^A03DischargePatient leaves the facility
ADT^A04RegisterPatient registers (outpatient, ED)
ADT^A08UpdateAny update to patient demographics
Which segments appear in each ADT type? 
ADT^A01 (Admit):     MSH  EVN  PID  PD1  NK1  PV1  PV2  IN1
ADT^A02 (Transfer):  MSH  EVN  PID            PV1
ADT^A03 (Discharge): MSH  EVN  PID            PV1  PV2
ADT^A04 (Register):  MSH  EVN  PID  PD1  NK1  PV1  PV2  IN1
ADT^A08 (Update):    MSH  EVN  PID  PD1  NK1  PV1  PV2  IN1

All share MSH + EVN + PID + PV1. The optional segments (PD1, NK1, PV2, IN1) vary by event type. Some segments like NK1 and IN1 can repeat (a patient may have multiple emergency contacts or insurance plans).

ORU — Observation Result (Lab Results)

Carries lab test orders and their results:

ORU^R01: MSH  PID  PV1  [ORC  OBR  OBX  OBX  OBX ...]
                         |---- repeating group ----|

The group in brackets repeats — one OBR per test ordered, with multiple OBX results per OBR.

SIU — Scheduling

SIU^S12 (New appointment): MSH  SCH  PID  RGS  AIL  AIP
Why This is a Data Cleaning Problem

In theory, HL7 messages follow the spec perfectly. In practice, they don’t.

Character-level corruption

Names, addresses, and IDs accumulate typos as data passes between systems:

SMITH^JOHN    (original)
SMTIH^JOHN    (transposition)
SMITH^JON     (deletion)
SMITH^JOHHN   (insertion)

These are the same patient, observed through noisy channels.

Structural variation

The HL7 spec defines a fixed number of fields per segment, but implementations often send fewer fields, truncating trailing empty ones. The number of fields present can vary across sending systems, creating structural “dialects” of the same segment type.

Missing and optional data

Fields are optional at multiple levels:

  • A segment may be absent from the message entirely (PD1, NK1, IN1 are optional in ADT)
  • A field within a present segment may be empty (consecutive ||)
  • Sub-components within a field may be missing (consecutive ^^)
Entity resolution

The same patient appears in thousands of messages over years of care. Each message carries a potentially different (corrupted) snapshot of their demographics. The core problem: given many noisy observations, recover the true patient identity and link records that refer to the same person.

Vocabulary drift

Code tables (sex, race, marital status) are supposed to use HL7-defined values, but implementations often use local codes, abbreviations, or free text. A “sex” field might contain M, MALE, 1, or Male depending on the source system.

Why This Matters for PPL Research

HL7v2 is a real-world domain with properties that align with open problems in probabilistic programming:

PropertyPPL challenge
Known generative structure (the spec defines the grammar) The model is derivable from the specification — not hand-crafted per dataset
Massive scale (millions of messages per institution) Inference must be tractable on large datasets
Multiple levels of corruption (character, structural, semantic) Requires composing noise models at different abstraction levels
Natural entity resolution (same patient across many messages) Latent entity inference with an unknown number of entities (CRP/DPM)
Recursive/variable-length structure (repeating segment groups) Recursive algebraic data types and geometric stopping distributions
